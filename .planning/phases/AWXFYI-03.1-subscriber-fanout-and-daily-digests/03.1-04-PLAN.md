# Plan 03.1-04: Monitor and Admin Digest Integration

## Goal

Wire subscriber fanout into the monitor/admin path so each monitor run summarizes new posts centrally, then sends one daily WhatsApp digest or no-change message per active subscriber channel with bounded operational visibility.

## Scope

This plan replaces Phase 3's immediate per-post alert behavior with daily digest/status delivery. It extends `MonitorRunResult` and admin DTOs/tests. It does not add the Phase 4 scheduler/CLI run-once behavior, Slack, Supabase UI, or retry automation.

## User Story

As the Airwallex FYI operator, I want `/admin/run-once` to discover and summarize new Airwallex posts, then send subscribers exactly one daily WhatsApp status message: a compact digest if updates exist, or a no-change message if none do.

## Artifacts This Plan Produces

- Monitor integration with `SubscriberSeedService` and `DailyDigestService`.
- Removal or bypass of immediate per-post WhatsApp alert sending from `handleNewPost`.
- Extended run result/admin DTO fields for digest delivered, no-change, duplicate-skip, and failed delivery counts/samples.
- Tests for new-post digest, no-change message, duplicate guard, approval-needed admin-only behavior, and subscriber failure isolation.

## Must Haves

### Truths

- New posts still auto-summarize through the existing canonical summary path.
- Delivery happens after monitor processing has produced or loaded eligible summaries.
- No-change daily messages are subscriber-facing and must be counted/sampled.
- Approval-needed items remain visible in run result/admin state, not in subscriber messages.
- Per-subscriber delivery status lives in digest delivery records, not only in post `processing_status`.
- Repeated same-day manual `/admin/run-once` calls do not send duplicate messages.

### Constraints

- Preserve first-run seed behavior and no historical spam.
- Preserve existing admin token protection.
- Keep `/admin/run-once` responses bounded.
- Do not add automatic retry or scheduling.

## Threat Model

<threat_model>
- Risk: immediate per-post send path survives and causes duplicate WhatsApp messages. Mitigation: tests assert multi-post day creates one combined send per channel.
- Risk: no-change messages spam on repeated manual runs. Mitigation: digest delivery unique local-date guard and duplicate-skip test.
- Risk: failed subscriber hides or blocks other sends. Mitigation: fanout records per-channel failures and continues; monitor result samples failures.
- Risk: approval-needed items leak to subscribers. Mitigation: digest service receives only eligible summarized posts; run-result approval samples stay admin-only.
- Risk: post status is abused for per-subscriber state. Mitigation: digest delivery records own per-subscriber status; posts keep canonical processing state.
</threat_model>
