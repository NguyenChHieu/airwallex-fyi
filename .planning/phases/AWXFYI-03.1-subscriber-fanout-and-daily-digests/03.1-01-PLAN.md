# Plan 03.1-01: Subscriber and Channel Persistence

## Goal

Add DB-backed subscriber and channel persistence so Phase 03.1 can fan out daily digest/status messages to active WhatsApp recipients without keeping recipients as config-only state.

## Scope

This plan creates the subscriber data model, repositories, config binding, and seed-once behavior. It does not send digest messages, create digest delivery records, or change `MonitorRunService` behavior yet.

## User Story

As the Airwallex FYI operator, I want my first WhatsApp recipient to be bootstrapped from config into the database, then managed as DB data so a future Supabase/admin surface can edit subscribers without code changes.

## Artifacts This Plan Produces

- `subscribers` table with display name, status, and timestamps.
- `subscriber_channels` table with subscriber FK, channel, recipient, status, and timestamps.
- Spring Data JDBC records/repositories for both tables.
- Seed-once service that creates the default WhatsApp subscriber/channel from config only when missing.
- Config defaults and tests for seed recipient/display name and service timezone.

## Must Haves

### Truths

- D-01 through D-05 must be represented in schema, config, and seed behavior.
- `WHATSAPP_TO` or the existing `airwallex-fyi.whatsapp.to` value remains the easy local setup path.
- DB rows are source of truth after initial seed; startup must not overwrite existing subscriber/channel rows.
- Supabase-friendly means simple UUID primary keys, explicit status strings, timestamps, and no nested JSON for core subscriber/channel fields.

### Constraints

- Do not edit `V1__initial_schema.sql`; add a new Flyway migration.
- Do not introduce a UI, Supabase dependency, or Slack/channel-specific abstractions beyond storing `channel` as a string.
- Do not move posts or summaries into subscriber tables.

## Threat Model

<threat_model>
- Risk: future Supabase/admin edits get overwritten by config. Mitigation: seed only when matching channel is missing; never sync/overwrite existing rows.
- Risk: duplicate WhatsApp channels receive duplicate digests. Mitigation: unique `(channel, recipient)` plus active-channel lookup.
- Risk: blank local config creates unusable delivery rows. Mitigation: seed only when recipient is non-blank.
</threat_model>
