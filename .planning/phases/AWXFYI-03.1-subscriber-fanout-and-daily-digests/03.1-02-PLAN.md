# Plan 03.1-02: Digest Delivery Persistence and Eligibility Contract

## Goal

Add the persistence layer for daily digest/status deliveries so the app can record one delivery attempt per subscriber channel per service-local day and preserve which posts were included in combined digest messages.

## Scope

This plan creates digest delivery tables, records, repositories, and eligibility helpers. It does not format WhatsApp messages or integrate digest fanout into `MonitorRunService` yet.

## User Story

As the Airwallex FYI operator, I want the database to know whether each subscriber channel already received today's digest/status message, and which posts were included, so manual reruns cannot spam subscribers and failures are auditable.

## Artifacts This Plan Produces

- `digest_deliveries` table with subscriber channel, local date, status, message type, payload preview, provider message ID/error, and timestamps.
- `digest_delivery_posts` table linking a digest delivery to included posts and summaries.
- Spring Data JDBC records/repositories for digest delivery persistence.
- Eligibility helper for summarized posts since the last successful digest delivery.
- Repository tests proving local-date uniqueness and post-link audit behavior.

## Must Haves

### Truths

- D-15 and D-16 are the core of this plan: delivery is digest-level, not per-post, and unique per active subscriber channel/local date.
- D-08 must be implemented as a query/service contract over canonical summaries and posts, not duplicate summary rows per subscriber.
- D-09 means no-change deliveries must be representable without linked posts.
- D-18 means failed rows are persisted, but automatic retry behavior is not added.

### Constraints

- Do not duplicate summary JSON into digest link rows; reference canonical `summaries`/`posts` where needed.
- Do not use `notification_attempts` as the daily digest table; it is per-post and cannot represent no-change status cleanly.
- Do not add scheduler behavior in this plan.

## Threat Model

<threat_model>
- Risk: repeated manual runs send duplicate same-day messages. Mitigation: unique `(subscriber_channel_id, local_date)` and lookup before sending.
- Risk: combined digest cannot be audited later. Mitigation: `digest_delivery_posts` links every included post/summary.
- Risk: summaries get copied per subscriber. Mitigation: link to canonical `summaries` rows; do not duplicate summary JSON.
- Risk: no-change message cannot be recorded because there is no post. Mitigation: digest delivery row supports message type `NO_CHANGES` with zero link rows.
</threat_model>
