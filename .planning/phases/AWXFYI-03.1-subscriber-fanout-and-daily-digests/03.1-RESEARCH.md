## Codebase Findings

### Existing Strengths

- `summaries` already has a unique `post_id`, so the canonical one-summary-per-post rule is already present.
- `MonitorRunService.runOnce()` is the correct integration point because it already knows which posts were seeded, new, updated, summarized, or approval-needed.
- `WhatsAppNotifier` already abstracts dry-run vs Twilio delivery, so digest fanout can reuse the provider boundary instead of adding another Twilio client.
- Admin responses already use bounded counts and sampled payloads/errors, which matches digest delivery visibility.
- Existing tests already use H2 in PostgreSQL mode, fake AI clients, fake notifiers, fixture HTML, and MockMvc.

### Current Gaps

- There is no subscriber or subscriber-channel table. The only recipient is `airwallex-fyi.whatsapp.to` in config.
- `notification_attempts` is per `post_id + channel + recipient`; it cannot represent one combined daily digest or a no-change message with no post.
- `WhatsAppAlertFormatter` formats one post at a time. Phase 03.1 needs a compact multi-post digest formatter and a no-change formatter.
- `MonitorRunService.handleNewPost()` currently sends immediately per new post. Phase 03.1 needs to summarize new posts centrally, then fan out one daily message per active subscriber channel.
- `MonitorRunResult` has summary/alert counts, but not subscriber/digest delivery counts, duplicate-skip counts, no-change counts, or per-subscriber delivery failures.
- No notification-specific repository tests exist yet, so Phase 03.1 should add focused subscriber/digest repository tests rather than relying on old coverage.

## Recommended Architecture

Use four vertical MVP slices:

1. Subscriber/channel persistence and seed-from-config behavior.
2. Digest delivery persistence with local-date uniqueness and delivery-post links.
3. Digest formatting and fanout service that reuses `WhatsAppNotifier`/Twilio transport boundaries.
4. Monitor/admin integration that replaces immediate per-post alerts with daily subscriber digest/status delivery.

## Planning Implications

- Add new Flyway migrations rather than editing `V1__initial_schema.sql`.
- Keep subscriber rows Supabase-friendly but minimal: display name, status, timestamps; channel, recipient, status, timestamps.
- Add service timezone config defaulting to `Australia/Sydney`.
- Use digest delivery records, not post status, as the source of truth for per-subscriber delivery. Post status can remain canonical post-processing state such as `SUMMARY_READY`.
- Preserve admin/run-once bounded response style with digest counts and samples.
- The no-change message is subscriber-facing and intentional, so roadmap/requirements must keep that behavior aligned.

## Validation Architecture

- Schema tests should verify `subscribers`, `subscriber_channels`, `digest_deliveries`, and `digest_delivery_posts` columns plus uniqueness/FK constraints.
- Repository/service tests should verify seed-once behavior does not overwrite DB-managed subscriber/channel edits.
- Digest tests should prove one local-date delivery guard prevents repeated manual runs from sending duplicate new-post or no-change messages.
- Formatter tests should verify compact multi-post digest text and exact no-change message text.
- Monitor tests should verify first no-change run sends one no-change message per active channel, new summarized posts produce one combined digest, approval-needed items are admin-only, and one subscriber failure does not block others.
- Final verification should run `./gradlew.bat test`.

## Risks And Mitigations

- Risk: config seeding overwrites future Supabase/admin changes. Mitigation: seed missing default channel once; do not sync/overwrite existing DB rows every startup.
- Risk: same-day manual reruns spam subscribers. Mitigation: unique `(subscriber_channel_id, local_date)` digest delivery guard.
- Risk: no-change messages conflict with earlier no-message default. Mitigation: CONTEXT.md and requirements explicitly supersede the old placeholder wording.
- Risk: one subscriber delivery failure blocks everyone. Mitigation: fanout loops per active subscriber channel, records individual status, and continues.
- Risk: combined digest loses per-post auditability. Mitigation: add digest-post link rows for every included post.
- Risk: post `processing_status` becomes ambiguous in multi-subscriber delivery. Mitigation: delivery tables own per-subscriber status; posts keep canonical processing state.

## Planning Recommendation

Proceed with four plans in dependency order:

- 03.1-01 Subscriber and channel persistence.
- 03.1-02 Digest delivery persistence and eligibility contract.
- 03.1-03 Digest formatting and subscriber fanout service.
- 03.1-04 Monitor/admin integration and regression coverage.
