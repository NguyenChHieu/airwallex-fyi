# Plan 03.1-03: Digest Formatting and Subscriber Fanout Service

## Goal

Build the service layer that formats one compact daily WhatsApp digest or no-change message, fans it out to active subscriber channels, records per-channel delivery status, and prevents duplicate same-day sends.

## Scope

This plan implements the digest formatter and fanout service. It may refactor the notifier interface enough to send digest messages without inventing fake post records. It does not yet wire the service into `MonitorRunService`; that happens in Plan 03.1-04.

## User Story

As a subscriber, I want one daily Airwallex FYI WhatsApp message: either a compact list of new summarized posts or a short no-change status, without duplicate messages when the operator reruns the monitor.

## Artifacts This Plan Produces

- Digest formatter for compact multi-post WhatsApp messages.
- Exact no-change message formatter.
- Daily digest fanout service over active subscriber channels.
- Duplicate guard using digest delivery local-date lookup.
- Per-subscriber delivery success/failure recording.
- Formatter, fanout, and notifier tests.

## Must Haves

### Truths

- D-11 through D-14 define the subscriber-facing message shape.
- D-16 through D-18 define duplicate and failure behavior.
- Approval-needed items must never appear in the digest body.
- The service must continue across subscriber-channel failures and record each failure.

### Constraints

- Keep Twilio details behind the existing notifier/transport boundary.
- Do not add Slack, chatbot, Supabase UI, or automatic retry behavior.
- Do not call live Twilio or Gemini in tests.

### Task 3: Add Daily Digest Fanout Service

</read_first>

<action>
- Add `DailyDigestService` that accepts the service-local date/time and eligible summarized posts.
- Load active WhatsApp subscriber channels.
- For each channel:
  - skip if a digest delivery already exists for that channel/local date.
  - format new-post digest when eligible posts exist; otherwise format no-change message.
  - send via notifier.
  - record delivery row with status, payload preview, provider message ID/error, and timestamps.
  - create delivery-post links only for new-post digest messages.
- Catch failures per subscriber channel, record a failed delivery, and continue to the next channel.
</action>

<acceptance_criteria>
- Multiple active subscriber channels each get one send attempt.
- A failed subscriber channel records failure and does not block successful channels.
- A second call on the same local date skips sends because delivery rows already exist.
- No-change messages create delivery rows with no delivery-post links.
</acceptance_criteria>

## Threat Model

<threat_model>
- Risk: same-day manual rerun sends duplicates. Mitigation: fanout checks digest delivery uniqueness before sending.
- Risk: one provider failure blocks all recipients. Mitigation: catch per channel, record failure, continue.
- Risk: digest body leaks operator approval-needed work. Mitigation: formatter accepts only eligible summarized posts.
- Risk: dry-run accidentally reaches Twilio after notifier refactor. Mitigation: dry-run implementation remains transport-free and tests assert fake transport is untouched.
</threat_model>
