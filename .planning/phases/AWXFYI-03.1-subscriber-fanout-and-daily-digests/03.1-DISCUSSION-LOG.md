## Subscriber Storage

| Option | Description | Selected |
|--------|-------------|----------|
| DB-backed subscribers table | Store subscribers/channels in Postgres, seed first WhatsApp recipient from config, and allow later admin/Supabase management. | yes |
| Config-only recipient list | Keep recipients in env/config for now; simpler, but weaker history/status and harder to manage later. | |
| Hybrid config plus DB | Env config creates/updates default subscribers in DB on startup; DB remains source of truth for delivery. | |

**User's choice:** DB-backed subscribers, configured so later Supabase can plug in.
**Notes:** User chose separate `subscribers` plus `subscriber_channels`, seed once from config, then manage in DB. Minimal admin-ready fields only.

## Digest Trigger

| Option | Description | Selected |
|--------|-------------|----------|
| During each monitor run, once per local day | Phase 4 can schedule the run daily; Phase 03.1 prevents repeated/manual runs from spamming subscribers. | yes |
| Separate digest job after monitoring | Cleaner separation, but more moving parts before scheduling exists. | |
| Immediate per-post alerts stay, plus daily digest later | Preserves current behavior, but conflicts with centralized daily updates and risks duplicate alerts. | |

**User's choice:** Send during monitor run with one-per-service-local-day guard.
**Notes:** User chose service-level timezone, default `Australia/Sydney`; include newly discovered successfully summarized posts since last successful digest; send a subscriber-facing no-change message if no eligible posts exist.

## Message Shape

| Option | Description | Selected |
|--------|-------------|----------|
| Compact multi-post digest | One message with title/date header, each post headline, 1-2 bullets, why it matters, and link. | yes |
| Current full alert repeated per post | Preserve Phase 3 detail level for each article, but busy days could get long/noisy. | |
| Headline/link only | Shortest message, but loses the AI summary value. | |

**User's choice:** Compact multi-post digest.
**Notes:** No-change message text is `Airwallex FYI: No new public Blog or Newsroom updates today.` Approval-needed items stay admin/run-result only. Multiple posts are combined into one daily WhatsApp message per subscriber channel.

---

## Delivery Dedupe

| Option | Description | Selected |
|--------|-------------|----------|
| Digest-level delivery table plus post links | Store one digest delivery per subscriber/channel/day, plus rows linking which posts were included. | yes |
| Reuse `notification_attempts` per post | Less schema work, but awkward for one combined daily message and no-change messages with no post. | |
| Only one table for digest deliveries | Simpler than links, but weaker audit trail for which posts were sent in each digest. | |

**User's choice:** Add digest-level delivery table plus post link rows.
**Notes:** Unique guard is one digest/status delivery per active subscriber channel per service-local date. Failed delivery for one subscriber does not block others. No automatic same-day retry in Phase 03.1.
