# Phase 03.1: Subscriber Fanout And Daily Digests - Context

**Gathered:** 2026-06-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 03.1 converts the Phase 3 single-recipient immediate WhatsApp alert path into a centralized subscriber fanout and daily digest model. The service should keep canonical posts and summaries service-owned, model subscribers and delivery channels separately, send one daily WhatsApp digest/status message per active subscriber channel per service-local day, and record per-subscriber delivery status so duplicate sends and one-recipient failures are handled cleanly.

Phase 03.1 does not add Slack delivery, chatbot interactions, a full Supabase/admin UI, production WhatsApp Business templates, automatic retry scheduling, or the unattended scheduler/CLI run-once hardening planned for Phase 4.

</domain>

<decisions>
## Implementation Decisions

### Subscriber Storage
- **D-01:** Use DB-backed subscribers, not a config-only recipient list. Config is only used to seed the first WhatsApp subscriber/channel for local setup.
- **D-02:** Model subscribers with separate `subscribers` and `subscriber_channels` tables. `subscribers` represents the person/subscription; `subscriber_channels` represents WhatsApp recipient/channel delivery targets.
- **D-03:** Seed the first WhatsApp subscriber/channel from config once, then treat Postgres as the source of truth. Do not overwrite later DB/Supabase/admin edits on every startup.
- **D-04:** Keep v1 fields minimal but admin-ready: subscriber `display_name`, `status`, `created_at`, `updated_at`; channel `channel`, `recipient`, `status`, `created_at`, `updated_at`.
- **D-05:** Keep the model Supabase-friendly so a later Supabase UI/API can manage subscribers without changing the digest delivery core.

### Digest Trigger
- **D-06:** Create/send digest output during each monitor run, but guard delivery so each active subscriber channel receives at most one digest/status message per service-local day. Phase 4 can schedule the monitor daily.
- **D-07:** Add a service-level timezone setting for the daily guard, defaulting to `Australia/Sydney`.
- **D-08:** New-post digests include newly discovered, successfully summarized posts since the last successful digest. Seeded/history posts and approval-needed content changes do not auto-send.
- **D-09:** If there are zero eligible new posts, send a subscriber-facing no-change WhatsApp message once per service-local day.
- **D-10:** Repeated manual runs on the same service-local day must not send duplicate new-post digests or duplicate no-change messages.

### Message Shape
- **D-11:** Use one compact multi-post WhatsApp digest message when there are new posts: title/date header, then each post with headline, 1-2 bullets, why it matters, and direct link.
- **D-12:** Use this no-change message text: `Airwallex FYI: No new public Blog or Newsroom updates today.`
- **D-13:** Keep approval-needed items, such as content-changed known URLs or missing summaries, in admin/run-result visibility only. Do not include them in subscriber digest messages.
- **D-14:** Send one combined WhatsApp message per active subscriber channel per day, not one message per post.

### Delivery Dedupe And Failure Handling
- **D-15:** Add a digest-level delivery table plus post link rows. The delivery table records one subscriber/channel/local-date message attempt; link rows record which posts were included when the digest contains posts.
- **D-16:** Use a uniqueness guard of one digest/status delivery per active subscriber channel per service-local date, regardless of whether the content is a new-post digest or a no-change message.
- **D-17:** If delivery fails for one subscriber channel, continue sending to other subscriber channels and record the failed delivery individually.
- **D-18:** Do not add automatic same-day retry in Phase 03.1. Store failed deliveries clearly so a later manual admin retry or future retry phase can be added.

### the agent's Discretion
- The planner may choose exact Kotlin package/class names, migration filenames, enum names, repository method names, and whether digest records store a rendered payload preview, as long as the decisions above stay true and tests cover the dedupe/failure behavior.
- The planner may decide whether to keep the existing `notification_attempts` table for per-post legacy alert attempts, replace the current Phase 3 send path with digest deliveries, or bridge both temporarily during implementation, as long as the resulting v1 user behavior is one daily subscriber message per channel.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `MonitorRunService.runOnce()` already has the right orchestration location after discovery, extraction, post state application, summarization, and notification visibility.
- `SummaryRecord` and `SummaryRepository` already enforce one canonical summary per post, which matches the centralized-summary decision.
- `WhatsAppNotifier` already hides Twilio/dry-run behavior behind an interface, so digest delivery can reuse the provider boundary.
- Existing H2/Flyway-backed repository tests and fake client patterns are suitable for subscriber, digest delivery, and dedupe tests.

### Established Patterns
- Package by capability under `com.airwallexfyi`.
- Keep external side effects behind small interfaces.
- Use environment-backed configuration and safe local defaults.
- Keep admin/run-once responses bounded with counts and samples rather than returning unbounded payloads.
- Prefer explicit relational constraints for dedupe over NoSQL flexibility.

### Integration Points
- Add subscriber/channel persistence and a seed-from-config path near application startup or digest service initialization.
- Replace immediate `handleNewPost` per-post alert sending with centralized digest selection and fanout after monitor processing knows which posts were newly summarized.
- Extend run results with digest delivery counts/samples, including no-change sends, skipped duplicate sends, and failed subscriber deliveries.
- Add digest delivery and delivery-post link tables through a new Flyway migration rather than rewriting `V1__initial_schema.sql`.

</code_context>

<specifics>
## Specific Ideas

- Future Supabase management matters: keep the schema simple and admin-friendly, but do not build the Supabase UI in this phase.
- Daily status behavior is intentional: subscribers should receive either a new-post digest or the no-change message once per service-local day.
- The no-change decision intentionally supersedes the earlier placeholder wording that said no subscriber message would be sent when no posts exist.
- Approval-needed items are operational/admin concerns, not subscriber-facing content.

</specifics>
