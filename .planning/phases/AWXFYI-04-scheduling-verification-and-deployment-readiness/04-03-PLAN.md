# Plan 04-03: Operational Docs and Environment Template

## Goal

Create the local-first operational documentation and safe environment template needed to run Airwallex FYI manually, on a schedule, and through the fixture-backed smoke check.

## Scope

This plan writes documentation only. It does not add Docker, CI/CD, platform-specific hosting files, Supabase wiring, Slack delivery, or real secrets.

## User Story

As the Airwallex FYI operator, I want clear commands and environment-variable examples so I can run tests, run one check, start scheduled polling, configure Twilio Sandbox/Gemini, and verify the app without guessing.

## Artifacts This Plan Produces

- `README.md` with local setup, test, run-once, scheduled-run, admin, Gemini, Twilio Sandbox, dry-run, and smoke-test sections.
- `.env.example` with every expected env var and safe placeholder values.

## Must Haves

### Truths

- D-13: Add local-first `README.md` with setup, test, run-once, scheduled-run, admin, Gemini, Twilio Sandbox, and smoke-test commands.
- D-14: Add `.env.example` with expected settings and safe placeholders only; no real secrets.
- D-15: Keep deployment guidance generic: long-running Spring Boot service with env vars, scheduler disabled by default, and `--run-once` for manual checks/smoke tests.
- D-16: Do not add Docker, Railway/Render/Fly.io, Supabase deployment wiring, or CI/CD automation in Phase 4 unless the user asks later.
- README must make dry-run and scheduler defaults clear.
- README must explain that automated smoke verification uses fixture/mocked data.
- `.env.example` must contain no real secrets.

### Constraints

- Do not add Docker/platform hosting docs unless the user chooses a target later.
- Do not tell users to commit real `.env` files.
- Do not claim live Twilio/Gemini sends happen in dry-run mode.
