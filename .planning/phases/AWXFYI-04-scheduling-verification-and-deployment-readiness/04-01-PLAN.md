# Plan 04-01: Runtime Run Modes

## Goal

Make the existing monitor runnable in the two operational modes required by Phase 4: one-shot command-line execution with `--run-once`, and opt-in scheduled polling controlled by existing scheduler configuration.

## Scope

This plan adds runtime glue around the existing monitor service. It does not change discovery, extraction, summarization, digest selection, subscriber fanout, admin authentication, or deployment documentation.

## User Story

As the Airwallex FYI operator, I want to run one monitor check from the command line and also run the service continuously on a schedule so I can test locally and later deploy it as a long-running service.

## Artifacts This Plan Produces

- `MonitorRunMode` helper that detects `--run-once` from Spring `ApplicationArguments`.
- `MonitorRunOnceCommand` service that calls `MonitorRunService.runOnce()` and maps result status to exit code.
- `RunOnceApplicationRunner` that invokes the one-shot command when `--run-once` is present.
- `ApplicationExit` abstraction so tests can assert exit code without terminating the test JVM.
- `MonitorScheduler` component using Spring scheduling and existing scheduler config.
- Scheduling enabled on the Spring Boot application.
- Unit tests for run-mode detection, one-shot command exit mapping, runner behavior, and scheduler guard behavior.

## Must Haves

### Truths

- D-01: Implement command-line `--run-once` as a Spring Boot one-shot runner that calls `MonitorRunService.runOnce()` directly, logs `MonitorRunResult`, then exits.
- D-02: Do not implement `--run-once` by calling `/admin/run-once`; CLI execution avoids admin-token, port, and self-HTTP complexity.
- D-03: `completed` exits with code `0`; `partial_failure` and `failed` produce non-zero exit codes.
- D-04: Scheduler, CLI run-once, and admin run-once all delegate to `MonitorRunService.runOnce()`.
- D-05: Unattended polling uses an in-app Spring scheduler controlled by `airwallex-fyi.scheduler.enabled` / `SCHEDULER_ENABLED`, default false.
- D-06: Use fixed-delay scheduling for MVP rather than cron/time-of-day scheduling.
- D-07: `--run-once` must not also start scheduled monitor execution in the same process.
- D-08: MVP assumes one running scheduler instance; DB/distributed locking is deferred.
- `MonitorRunService.runOnce()` remains the only monitor orchestration path.
- `/admin/run-once` remains untouched except for continuing to share the same monitor service.
- `SCHEDULER_ENABLED=false` remains the default safe local behavior.

### Constraints

- Do not add a second monitor orchestration service.
- Do not call the admin HTTP endpoint from command-line mode.
- Do not add cron scheduling or distributed locking in this phase.
- Do not make tests call real Airwallex, Gemini, or Twilio.

<action>
- Add package `com.airwallexfyi.runtime`.
- Add `MonitorRunMode` that accepts Spring `ApplicationArguments` and exposes `runOnce: Boolean` true when `--run-once` is present.
- Add `MonitorRunOnceCommand` with an `execute(): Int` method that calls `MonitorRunService.runOnce()` exactly once, logs the result status/message/counts, and returns `0` only when `result.status == MonitorRunStatus.COMPLETED`; return `1` for `partial_failure` or `failed`.
- Add `ApplicationExit` with a production implementation that terminates the Spring application with the provided exit code, but keep the interface small enough to fake in tests.
- Add `RunOnceApplicationRunner` that does nothing unless `MonitorRunMode.runOnce` is true; when true, it calls `MonitorRunOnceCommand.execute()` and passes the returned code to `ApplicationExit.exit(code)`.
</action>

- `MonitorRunOnceCommandTest` proves `partial_failure` and `failed` map to exit code `1`.
- `RunOnceApplicationRunnerTest` proves the runner does not call the monitor or exit abstraction when `--run-once` is absent.
- `RunOnceApplicationRunnerTest` proves the runner calls the command and exit abstraction when `--run-once` is present.
</acceptance_criteria>

## Threat Model

<threat_model>
- Risk: `--run-once` accidentally sends duplicate subscriber messages because scheduler also fires. Mitigation: `MonitorRunMode.runOnce` guard inside scheduler plus process exit after one-shot command.
- Risk: scripts cannot detect failed monitor checks. Mitigation: `MonitorRunOnceCommand` maps non-completed statuses to exit code `1`.
- Risk: local development unexpectedly polls or sends. Mitigation: scheduler remains disabled by default and dry-run remains true by default.
</threat_model>
