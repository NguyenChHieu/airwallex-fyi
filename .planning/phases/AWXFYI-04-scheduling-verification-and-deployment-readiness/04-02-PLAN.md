# Plan 04-02: Fixture-Backed Smoke and Coverage Closure

## Goal

Add the missing operational verification for Phase 4 and prove the full dry-run monitor path can run against local fixture or mocked Airwallex data without live external calls.

## Scope

This plan adds or tightens tests only. It does not add production fixture modes, change provider credentials, or alter subscriber/digest behavior unless a test exposes a real wiring bug.

## User Story

As the Airwallex FYI operator, I want one repeatable command that proves the service can run the monitor pipeline in dry-run mode with local data before I deploy or schedule it.

## Artifacts This Plan Produces

- `RunOnceSmokeTest` or equivalent Spring Boot smoke test for the one-shot monitor command.
- Coverage confirmation for scheduler/run-once behavior from Plan 04-01.
- Focused assertions showing QUAL-01 coverage remains intact across sitemap filtering, extraction, seed, dedupe, dry-run notification, and new-post digest flow.

## Must Haves

### Truths

- D-09: Automated verification stays fixture-backed or mocked; no live Airwallex, Gemini, or Twilio network calls.
- D-10: Tests cover scheduler enabled/disabled behavior, CLI `--run-once` behavior, status signaling, and dry-run smoke flow.
- D-11: Reuse Spring Boot tests, H2 in PostgreSQL mode, fixture HTML/sitemap data, fake source/AI/notifier clients, and bounded assertions.
- D-12: Do not rewrite already-covered tests unless needed; current tests cover sitemap filtering, extraction, seed, dedupe, dry-run/no-change digest, new-post digest, delivery failure isolation, and content-change approval.
- The smoke test must use fixture/mocked Airwallex data and fake AI/notifier dependencies.
- The smoke test must prove Twilio is not called.
- Existing high-value tests must be reused rather than duplicated wholesale.

### Constraints

- Do not call the live Airwallex sitemap in automated tests.
- Do not require `GEMINI_API_KEY`, `TWILIO_ACCOUNT_SID`, `TWILIO_AUTH_TOKEN`, or real WhatsApp numbers for automated tests.
- Do not add production-only fake-data switches unless documentation explicitly marks them test-only.
