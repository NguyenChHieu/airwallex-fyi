# Plan 02-04: Admin Run Visibility and Recent Post Filters

## Goal

Expose Phase 2 source-monitoring behavior through the existing admin API so manual runs and recent-post inspection show accurate operational status without leaking excessive article bodies or noisy internal errors.

## Scope

This plan updates admin API DTOs and query behavior only. It does not add a UI, scheduler, summarization, or notifications.

## Artifacts This Phase Produces

- `/admin/run-once` response mapped from real `MonitorRunResult` counts and samples.
- `/admin/posts/recent` filters for source type and processing status.
- Short body previews instead of full article bodies.
- Admin controller tests covering success, partial failure, filters, preview truncation, and auth preservation.

## Must Haves

### Truths

- D-19: `/admin/run-once` returns operational counts and small samples.
- D-20: Run statuses are explicit: `completed`, `partial_failure`, and `failed`.
- D-21: `/admin/posts/recent` gets simple `sourceType` and `processingStatus` filters if cleanly scoped.
- D-22: Admin responses show short body previews only.
- D-23: Detailed errors remain in logs; responses include sampled errors only.
- D-24: Phase 2 has no dry-run mode for discovery or extraction.
- D-14: Admin output must make first-run seed behavior visible through counts.
- D-15: Admin output must not label existing changed URLs as new.
- D-16: Admin output should expose updated counts for known URL hash changes.
- D-17: Admin output should expose sampled article extraction failures while the run continues.

### Constraints

- Preserve existing admin token protection.
- Do not return full `article_body` from list endpoints.
- Keep query filters simple; use a small query service if repository derived methods become awkward.
- Do not introduce a public unauthenticated endpoint.

## Threat Model

<threat_model>
- Risk: admin endpoint leaks too much scraped content. Mitigation: return body previews only.
- Risk: detailed exception messages leak internals. Mitigation: log detailed errors but return sampled short reasons.
- Risk: new admin parameters accidentally bypass auth. Mitigation: preserve the existing token filter and controller security tests.
</threat_model>
