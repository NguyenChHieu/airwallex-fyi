# Plan 02-03: Monitor Run Persistence, Dedupe, and Seed Behavior

## Goal

Wire discovery and extraction into the monitor run so the app can seed recent public updates, detect truly new URLs, quietly update known changed content, and continue through partial extraction failures.

## Scope

This plan implements database state changes for monitor runs. It does not summarize with AI, send Slack or WhatsApp messages, or build a scheduler beyond keeping the run service callable by later scheduled jobs.

## Artifacts This Phase Produces

- `MonitorRunResult` with operational status, counts, samples, and external-call flags.
- `PostStateService` for insert/update/dedupe behavior.
- `MonitorRunService.runOnce()` orchestration over discovery, extraction, and persistence.
- Tests covering first-run seed, repeat runs, content updates, extraction failures, and discovery failures.

## Must Haves

### Truths

- D-04: Sitemap fetch failure keeps the service usable, reports discovery failure, and writes no new posts.
- D-14: First run seeds only a configurable recent baseline; default is 25 most recent posts.
- D-15: New means URL not in DB. Existing URLs are not new even if content changes.
- D-16: Known URL content hash changes update metadata, body, and hash quietly; they are counted as updated, not new.
- D-17: Per-article extraction failure reports URL/reason, skips that post, and continues.
- D-18: Keep Postgres/Spring Data JDBC; do not switch to MongoDB.
- D-20: Run statuses are explicit: `completed`, `partial_failure`, and `failed`.
- Phase 2 must not send external notifications or call AI models.

### Constraints

- Preserve the existing database schema unless a focused migration is truly required.
- Use relational uniqueness on URL for dedupe; do not rely only on in-memory filtering.
- Be careful with Spring Data JDBC `Persistable` semantics. Existing rows may need explicit update queries via `JdbcClient` or `NamedParameterJdbcTemplate` rather than blindly calling `save()`.
- Keep article failure details sampled in run results and detailed in logs.

## Threat Model

<threat_model>
- Risk: duplicate notifications later because existing changed posts are treated as new. Mitigation: URL existence defines newness; content changes count as updates.
- Risk: first run floods later notification channels. Mitigation: seed only the most recent configurable baseline and mark seeded rows separately if existing status model supports it.
- Risk: failed article breaks the whole daily check. Mitigation: catch per-article extraction failures, sample errors, and continue.
- Risk: Spring Data JDBC save semantics insert duplicate rows. Mitigation: use explicit update SQL for known rows and keep URL uniqueness enforced.
</threat_model>
