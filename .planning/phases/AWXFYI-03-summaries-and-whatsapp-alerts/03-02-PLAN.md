# Plan 03-02: Gemini Strict JSON Summarizer

## Goal

Implement the Gemini-backed summary provider and summary service so the app can turn a newly discovered article into one validated `StructuredSummary` and persisted summary row.

## Scope

This plan calls or fakes Gemini only. It does not format WhatsApp messages, call Twilio, or integrate the summarizer into `MonitorRunService` yet.

## User Story

As the Airwallex FYI operator, I want new article text summarized through a free hosted AI provider so that the alert pipeline has useful structured content without hand-written summaries.

## Artifacts This Phase Produces

- Provider-neutral AI/Gemini configuration.
- `AiSummaryClient` interface.
- Gemini REST client using structured JSON output request shape.
- Strict parser/validator that blocks invalid provider output.
- `ArticleSummaryService` that saves successful summaries and reports summary failures without notifications.
- Unit tests with fake Gemini transport; no live Gemini calls in tests.

## Must Haves

### Truths

- D-01: Gemini free tier first, behind a provider-backed summarizer interface.
- D-02: Strict JSON fields are required.
- D-03: Invalid Gemini output records failure and blocks WhatsApp in later plans.
- D-05: Use current official Gemini structured-output docs; do not assume API behavior from memory.
- AI-01: Only posts that need a summary should reach the AI client.

### Constraints

- Use Spring `RestClient` or a small transport wrapper; avoid adding a Gemini SDK dependency unless implementation proves it is clearly simpler.
- Keep `GEMINI_API_KEY` environment-backed and never log it.
- Keep `AI_MODEL` configurable; default may follow the verified docs, but tests must not depend on a live model.
- No live network calls in tests.

## Threat Model

<threat_model>
- Risk: Gemini output is trusted too much. Mitigation: parse and validate JSON locally before saving or notifying.
- Risk: API keys leak in logs/errors. Mitigation: never include `GEMINI_API_KEY` in exception messages or admin samples.
- Risk: provider model changes break output shape. Mitigation: keep model configurable and tests fake the provider response contract.
- Risk: summary prompt sends unnecessary data. Mitigation: send only title, description, source type, URL, and normalized article body from public Airwallex pages.
</threat_model>
