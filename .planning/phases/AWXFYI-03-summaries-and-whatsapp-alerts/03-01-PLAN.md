# Plan 03-01: Summary Persistence and Lifecycle Status Contract

## Goal

Create the strict local summary contract, summary persistence mapping, and coarse post lifecycle statuses that later Gemini and WhatsApp work can use.

## Scope

This plan does not call Gemini or Twilio. It creates the local data contract and persistence layer used by later Phase 3 plans.

## User Story

As the Airwallex FYI operator, I want the service to store one validated structured summary per post so that later alert delivery and admin inspection can use a stable contract.

## Artifacts This Phase Produces

- `StructuredSummary` domain type with headline, bullets, why-it-matters, tags, and source type.
- Spring Data JDBC summary persistence mapped to the existing `summaries` table.
- `SummaryRepository` with lookup by `postId`.
- Expanded `ProcessingStatus` enum for Phase 3 lifecycle states.
- Persistence tests proving summary rows and status values work against the Flyway schema.

## Must Haves

### Truths

- D-02: Summary fields are `headline`, `bullets`, `why_it_matters`, `tags`, and `source_type`.
- D-04: Summary input should use extracted text fields, not images/media.
- D-18: Post status gets simple lifecycle values: `SUMMARY_READY`, `ALERT_SENT`, `DRY_RUN_READY`, `SUMMARY_FAILED`, `ALERT_FAILED`, and `APPROVAL_NEEDED`.
- D-19: Status values stay coarse and admin-friendly.
- AI-01: Summary output must be persisted for posts that need a new summary.

### Constraints

- Use the existing `summaries` table from `V1__initial_schema.sql` unless a test proves a migration is required.
- Keep the Kotlin persistence model Spring Data JDBC friendly.
- Preserve the existing `DISCOVERED` and `SEEDED` statuses.
- Do not add Gemini/Twilio dependencies in this plan.
