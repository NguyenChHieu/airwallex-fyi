# Plan 03-03: WhatsApp Alert Formatter, Dry-Run, and Twilio Notifier

## Goal

Build the deterministic WhatsApp alert payload, dry-run notifier, Twilio Sandbox notifier, and notification-attempt persistence needed to deliver or preview Phase 3 alerts.

## Scope

This plan does not integrate notification sending into `MonitorRunService`; Plan 03-04 wires the formatter/notifier into the monitor pipeline.

## User Story

As the Airwallex FYI operator, I want a new article summary to become an inspectable WhatsApp alert payload and, when dry-run is disabled, to send through Twilio Sandbox.

## Artifacts This Phase Produces

- `WhatsAppAlertPayload` representing channel, recipient, body, source URL, and preview text.
- `WhatsAppAlertFormatter` with deterministic detailed message format.
- `WhatsAppNotifier` interface.
- `DryRunWhatsAppNotifier` that records/skips sends without Twilio.
- `TwilioWhatsAppNotifier` using Twilio Messages REST endpoint shape.
- Active `NotificationAttemptRecord` and `NotificationAttemptRepository` mapped to existing schema.
- Unit tests proving formatting, dry-run safety, and Twilio request construction.

## Must Haves

### Truths

- D-11: Detailed but concise WhatsApp format.
- D-12: Direct source link is mandatory.
- D-13: Formatter is separate from summarizer and sender.
- D-14: Dry-run builds the exact payload and records status.
- D-15: Dry-run never calls Twilio.
- D-16: Twilio Sandbox send path sits behind a notifier interface.
- D-17: Twilio failure is recorded, not retried.
- NOTIF-01: Twilio Sandbox send path works behind an interface.
- NOTIF-02: Dry-run logs/returns payload and avoids Twilio.

### Constraints

- Use existing config values for Twilio account SID, auth token, WhatsApp from, recipient, and dry-run.
- Do not log Twilio auth token.
- Do not add retry scheduling in Phase 3.
- Keep tests fake; no live Twilio calls.

### Task 4: Add Twilio Sandbox Notifier Behind Interface

<read_first>
- `src/main/kotlin/com/airwallexfyi/sources/AirwallexHttpClient.kt`
- `src/main/kotlin/com/airwallexfyi/config/AppProperties.kt`
- `.planning/phases/AWXFYI-03-summaries-and-whatsapp-alerts/03-RESEARCH.md`
</read_first>

<action>
- Add `TwilioWhatsAppNotifier` that uses a fakeable transport or Spring `RestClient` wrapper.
- Build a `POST https://api.twilio.com/2010-04-01/Accounts/{accountSid}/Messages.json` request.
- Send form parameters `From`, `To`, and `Body` where `From` and `To` are WhatsApp channel addresses such as `whatsapp:+14155238886`.
- Use Basic Auth with account SID and auth token, but never log the token.
- On successful response, save notification attempt status `SENT`, provider message ID from the returned SID, and sent timestamp.
- On provider exception or failed response parsing, save notification attempt status `FAILED` with bounded error text and return failure without retry.
</action>

<acceptance_criteria>
- Twilio notifier request contains `From`, `To`, and `Body` values from config/payload.
- Successful fake Twilio response records `provider_message_id` and status `SENT`.
- Failed fake Twilio response records status `FAILED`, bounded error text, and no retry attempt.
- Auth token never appears in assertion snapshots, logs, or returned error strings.
</acceptance_criteria>

<verify>
- `./gradlew.bat test --tests "*TwilioWhatsAppNotifierTest"`
</verify>

## Threat Model

<threat_model>
- Risk: dry-run accidentally sends a real WhatsApp message. Mitigation: dry-run notifier has no Twilio dependency and tests assert no Twilio transport is called.
- Risk: duplicate alerts for the same post/recipient. Mitigation: notification lookup and existing unique constraint block duplicate attempts.
- Risk: Twilio credentials leak. Mitigation: no token in logs, results, or errors; tests check sanitized failure text.
- Risk: message format becomes impossible to test. Mitigation: pure formatter with exact output tests.
</threat_model>
